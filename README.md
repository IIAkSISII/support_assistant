# support-assistant

`support-assistant` - микросервис бота первой линии техподдержки для задания "Боты и автоматизация для СпросиИИ". Сервис принимает webhook с новым сообщением, определяет категорию и срочность обращения, подбирает готовый ответ из базы знаний и, если нужно, формирует контекст для эскалации оператору.

## Идея проекта

Проект реализует сценарий из задания: автоматизировать первичную обработку обращений поддержки без изменения основного продукта. Бот:

- принимает входящие события `message_created`;
- отбрасывает неподходящие события, исходящие и приватные сообщения;
- хранит историю диалога в Redis для передачи контекста в анализатор;
- отправляет сообщение, историю и базу знаний в LLM;
- получает классификацию: `category`, `priority`, `keywords`, `escalate`;
- ищет готовый ответ в JSON-базе знаний;
- возвращает итоговый `ProcessResult` и при необходимости добавляет `operator_context` для оператора;
- опционально отправляет результат обратно во внешнюю систему через API Chatwoot.

## Как устроен сервис

Основные части проекта:

- [cmd/app/main.go](cmd/app/main.go) - запуск HTTP-сервера, сборка зависимостей, graceful shutdown.
- [internal/handler/webhook.go](internal/handler/webhook.go) - HTTP-обработчик `POST /webhook`.
- [internal/service/processor.go](internal/service/processor.go) - основная бизнес-логика обработки сообщения.
- [internal/llm/analyzer.go](internal/llm/analyzer.go) - клиент к LLM через OpenAI-compatible `/chat/completions`.
- [internal/repository/knowledge/knowledge.repository.go](internal/repository/knowledge/knowledge.repository.go) - загрузка и поиск ответов в JSON-базе знаний.
- [internal/repository/history/redis.repository.go](internal/repository/history/redis.repository.go) - хранение истории диалогов в Redis.
- [internal/client/chatwoot.go](internal/client/chatwoot.go) - отправка результата обратно в Chatwoot.
- [docs/swagger.yaml](docs/swagger.yaml) - описание API Swagger.

## Требования

- Go `1.25+` для локального запуска
- Docker и Docker Compose для контейнерного сценария
- Доступ к языковой модели: локально через Ollama или через внешний API, например DeepSeek

Сервис отправляет запросы к языковой модели в формате OpenAI Chat Completions. Поэтому можно использовать разные совместимые варианты:

- DeepSeek API;
- локальный Ollama;
- vLLM;
- другой сервис, который поддерживает `POST /chat/completions`.

## Переменные окружения

Базовый шаблон лежит в [.env.example](/E:/GolangProjects/support-assistant/.env.example).

Основные переменные:

- `HTTP_ADDR` - адрес HTTP-сервера, по умолчанию `:8090`
- `KNOWLEDGE_BASE_PATH` - путь к JSON-базе знаний
- `LLM_API_KEY` - API key для внешнего провайдера или любой непустой токен для Ollama
- `LLM_BASE_URL` - базовый URL LLM API, например `http://ollama:11434/v1`
- `LLM_MODEL` - модель для анализа обращений
- `LLM_MAX_TOKENS` - максимальное количество токенов, которое модель может сгенерировать в ответе
- `LLM_TIMEOUT_SECONDS` - таймаут запроса к модели
- `HISTORY_LIMIT` - сколько последних сообщений передавать в анализатор
- `REDIS_ADDR` - адрес Redis, например redis:6379 внутри Docker Compose или localhost:6379 при локальном запуске
- `REDIS_PASSWORD` - пароль Redis, если он настроен. Оставьте пустым, если Redis запущен без пароля.
- `REDIS_DB` - номер логической базы Redis, по умолчанию 0
- `REDIS_KEY_PREFIX` - префикс ключей Redis для истории диалогов
- `REDIS_OPERATION_TIMEOUT_SECONDS` - таймаут операций Redis
- `CHATWOOT_ENABLED` - включить/выключить отправку результата обратно в Chatwoot
- `CHATWOOT_BASE_URL` - адрес Chatwoot API
- `CHATWOOT_API_ACCESS_TOKEN` - токен Chatwoot
- `LOG_LEVEL` — уровень логирования: `debug`, `info`, `warn`, `error`.
- `LOG_FORMAT` — формат логов: `json` или `text`.

## Запуск

### 1. Локально через Go

Скопируйте шаблон окружения и при необходимости поправьте значения:

```powershell  
Copy-Item .env.example .env  
```  

Для локального запуска переменные нужно либо экспортировать в текущую PowerShell-сессию, либо задать вручную. Один из вариантов:

```powershell  
Get-Content .env | ForEach-Object {  
  if ($_ -match '^\s*$' -or $_ -match '^\s*#') { return }  $name, $value = $_ -split '=', 2  Set-Item -Path "Env:$name" -Value $value}  
```  

Если используется локальный `Ollama`, достаточно оставить `LLM_BASE_URL=http://localhost:11434/v1`, выбрать установленную модель и задать любой непустой `LLM_API_KEY`.

Запуск:

```powershell  
go run ./cmd/app  
```  

Сервис поднимется на `http://localhost:8090`.

### 2. Через Docker Compose

В репозитории уже есть готовый сценарий с `api`, `ollama` и отдельным сервисом для предварительной загрузки модели. Запуск:

```powershell  
docker compose up --build  
```  

После старта:

- API будет доступен на `http://localhost:8090`
- Swagger UI будет доступен на `http://localhost:8090/swagger/index.html`

При первом запуске модель может скачиваться заметное время. Она сохраняется в Docker volume `ollama-data`, поэтому при последующих запусках повторно скачиваться не должна.

### 3. Проверка API вручную

Основная точка входа:

- `POST /webhook`

Минимальный пример запроса:

```json  
{  
"event": "message_created",  
"id": 123,  
"content": "Я оплатил подписку, но доступ не появился",  
"message_type": "incoming",  
"private": false,  
"sender": {  
"id": 456  
},  
"account": {  
"id": 1  
},  
"conversation": {  
"id": 789  
},  
"inbox": {  
"id": 10,  
"name": "Support"  
},  
"attachments": []  
}  
```  

В ответ сервис возвращает JSON с категорией, приоритетом, ответом пользователю и флагом эскалации. Если эскалация нужна, в ответе появляется `operator_context`.

## Логи и диагностика

Сервис пишет структурированные логи через `slog`.

- для локального запуска логи идут в stdout процесса;
- для Docker удобно смотреть их так:

```powershell  
docker compose logs -f api  
```  

Сервис логирует:

- запуск HTTP-сервера;
- обработку HTTP-запросов;
- причины игнорирования webhook-событий;
- ошибки валидации;
- ошибки LLM;
- результат классификации;
- факт эскалации.
- потребление ресурсов HTTP-запроса: размер ответа, heap-память, аллокации за время обработки, GC delta и количество goroutine;
- длительность LLM-запроса и token usage, если выбранный OpenAI-compatible провайдер возвращает `usage`.

Если запускать `docker compose up` без `-d`, Docker Compose будет показывать логи всех сервисов сразу. В таком режиме логи `ollama` могут мешать читать логи `api`. Для отладки приложения удобнее запускать контейнеры в фоне.

## Что покрыто по функциональности

- приём webhook-событий `message_created`;
- фильтрация неподходящих событий, исходящих и приватных сообщений;
- валидация входящего webhook-запроса;
- классификация обращений через LLM;
- определение категории обращения;
- определение ключевых слов;
- определение приоритета: `low`, `medium`, `high`, `critical`;
- определение необходимости эскалации оператору;
- учёт последних сообщений из истории диалога при анализе обращения;
- JSON-база знаний с готовыми ответами;
- подбор готового ответа по категории и ключевым словам;
- подготовленный ответ для уточнения непонятного обращения;
- подготовленный ответ для передачи обращения оператору;
- подготовка `operator_context` при эскалации;
- передача оператору краткого summary, причины эскалации, рекомендуемого действия и истории диалога;
- взаимодействие `api` с локальной моделью через внутреннюю Docker-сеть;
- взаимодействие api с Redis через внутреннюю Docker-сеть;
- опциональная отправка результата обработки во внешний Chatwoot;
- структурированные логи через `slog`;
- Swagger-документация;
- контейнерный запуск через Docker Compose;
- отдельный Docker Compose profile для запуска тестов.

## Ограничения текущей реализации

- база знаний статическая и хранится в JSON-файле;
- изменение базы знаний требует обновления файла и перезапуска сервиса;
- качество классификации зависит от выбранной модели, промпта и содержимого базы знаний;
- локальная модель может отвечать медленно на слабом железе.
